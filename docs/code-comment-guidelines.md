# 代码注释规范

## 1. 目的

这份规范用于统一本仓库的源码注释风格，解决下面几个常见问题：

- 文件有没有写清楚“这个文件是干什么的”
- 核心类、函数、方法有没有写清楚输入、输出和副作用
- 注释是不是只是重复代码字面意思
- 注释能不能在后续迭代里持续维护，而不是写完一次就失效

本规范默认适用于：

- Python 源码：`rag/`、`scripts/`、`detection/`、`vision/`、`robotics/`、`apps/mac/desktop/`
- Swift 源码：`apps/mac/mac-app/`
- Shell 脚本：`scripts/*.sh`

## 2. 基本原则

- 注释优先回答“为什么这样设计”和“这一层负责什么”，不要只翻译代码表面含义。
- 文件头注释写文件职责、边界和主要入口，不在文件头罗列全部参数。
- 参数、返回值、异常、线程语义、副作用，优先写在函数、方法、类的注释里。
- 简单、显而易见的代码不强制写注释；复杂逻辑、跨模块协作、隐含约束必须写。
- 注释必须和代码一起维护。改了行为但不改注释，视为注释失效。

## 3. 必须写注释的地方

### 3.1 文件级注释

下列文件必须在顶部提供文件级说明：

- 入口文件：如 API 入口、CLI 入口、桌面应用入口
- 服务编排文件：如聚合多个模块的 service、pipeline、view model
- 核心流程文件：如构建知识库、训练、推理、数据转换
- 对外接口文件：如 HTTP client、web service、数据存储接口
- 需要阅读上下文才能理解的脚本文件

文件级注释至少应说明：

- 这个文件负责什么
- 这个文件不负责什么，或者边界在哪里
- 主要入口类、入口函数或核心对象是谁

### 3.2 类型级注释

下列类型必须写注释：

- 对外暴露的 `class`、`struct`、`enum`
- 业务核心类型
- 状态容器、服务对象、配置对象
- 命名无法完整表达职责的类型

类型级注释至少应说明：

- 该类型承担的职责
- 关键状态或生命周期
- 重要约束，例如线程限制、依赖关系、可变性边界

### 3.3 函数/方法级注释

下列函数或方法必须写注释：

- 公共函数、公共方法
- 有参数约束、返回约束、异常约束的方法
- 有副作用的方法，例如网络请求、文件写入、数据库更新、UI 状态变更
- 业务规则集中点
- 不容易从函数名直接看懂的逻辑

函数/方法级注释优先说明：

- 参数含义
- 返回值含义
- 可能抛出的错误或失败方式
- 副作用
- 调用前提和调用后的状态变化

## 4. 可以不写或少写的地方

下列情况可以不强制写注释，避免形式主义：

- 很短且语义直接的私有 helper
- 纯数据搬运、没有业务判断的简单 getter / setter
- 单行常量定义，且命名已足够清晰
- 明显的 UI 拼装代码，除非存在复杂状态约束或交互意图

判断标准不是“有没有注释”，而是“读者能不能在几秒内理解这段代码的职责和约束”。

## 5. 文件级注释写什么，不写什么

### 5.1 应该写

- 文件角色：例如“本文件负责组装 RAG 检索与生成链路”
- 上下游关系：例如“上游接 FastAPI，请求下游调用 Retriever 和 Generator”
- 边界：例如“本文件不负责模型训练，只负责在线推理编排”

### 5.2 不应该写

- 把文件里的每个函数再抄一遍
- 把每个参数都堆在文件头
- 与代码完全重复的描述，例如“这个函数用于返回结果”
- 会频繁过期的实现细节，例如临时路径、当前调试用常量

## 6. 各语言建议格式

### 6.1 Python

文件级说明使用模块 docstring，放在文件最上方。

示例：

```python
"""构建知识库索引并写入缓存产物。

本模块负责加载源文档、清洗文本、切块、计算向量并写出索引文件。
它不负责在线问答服务，只负责离线索引构建。
"""
```

公共类和函数优先使用 Google 风格 docstring。

```python
def build_index(source_dir: Path, output_dir: Path, force: bool = False) -> None:
    """构建知识库索引。

    Args:
        source_dir: 原始资料目录。
        output_dir: 索引与元数据输出目录。
        force: 为 True 时覆盖已有产物。

    Raises:
        FileNotFoundError: 当输入目录不存在时抛出。
    """
```

### 6.2 Swift

文件顶部使用简短注释块说明文件职责，公共类型和重要方法使用 `///`。

```swift
// 管理 mac app 的主状态，包括问答、导入、检测和训练数据相关流程。
// 本文件负责 UI 状态编排，不直接承担底层网络传输和 Python 进程实现。

@MainActor
final class AppViewModel: ObservableObject {
    /// 导入用户选择的资料文件并刷新知识库状态。
    /// - Parameter urls: 用户从文件选择器选中的本地文件列表。
    func importDocuments(from urls: [URL]) async { ... }
}
```

如果方法有线程或 actor 约束，应在注释中明确写出。

### 6.3 Shell

Shell 脚本至少应包含：

- shebang
- 一行脚本用途说明
- 关键环境变量或前置依赖说明

示例：

```bash
#!/usr/bin/env bash
# 构建并运行本地 SwiftUI mac app，要求当前目录位于仓库根目录。
```

## 7. 推荐模板

### 7.1 Python 文件模板

```python
"""一句话说明文件职责。

补充说明这个文件的边界、主要入口和上下游关系。
"""
```

### 7.2 Python 函数模板

```python
def func(arg1: str, arg2: int) -> bool:
    """一句话说明函数作用。

    Args:
        arg1: 参数 1 的业务含义。
        arg2: 参数 2 的业务含义。

    Returns:
        返回值代表的业务语义。

    Raises:
        ValueError: 输入不合法时抛出。
    """
```

### 7.3 Swift 类型模板

```swift
/// 一句话说明这个类型的职责。
/// 如有必要，补充线程约束、生命周期和依赖关系。
struct ExampleType {
}
```

## 8. 反例

下面这些写法不符合本规范：

- `# 初始化变量`
- `# 调用接口`
- `# 返回结果`
- `/// This is a view model`
- 文件头写成长篇历史背景，但没有说明当前职责和边界
- 文件头堆满参数说明，但函数本身没有 docstring

这些问题的共同点是信息密度低，读者看完仍然不知道真正的接口约束和业务角色。

## 9. 评审检查清单

提交代码时，至少检查下面几项：

- 文件头是否能在 10 秒内说明这个文件做什么
- 公共接口是否写清楚参数、返回值和失败方式
- 是否把“为什么”和“边界”写清楚，而不是只复述代码
- 注释是否仍然和当前实现一致
- 是否只在必要处写注释，没有制造大量无效噪音

## 10. 本仓库落地策略

为了避免一次性大面积补注释导致噪音过高，建议按优先级推进：

1. 先补入口文件、service、pipeline、view model、核心脚本。
2. 再补公共类、公共函数和跨模块接口。
3. 最后按模块逐步清理剩余高价值文件。

当前仓库优先建议补齐的文件类型：

- `rag/api/main.py`、`rag/pipeline.py`、`rag/generator.py`、`rag/retriever.py`
- `scripts/build_kb.py`、`scripts/source_loader.py`、`scripts/chunk_splitter.py`
- `detection/service.py`、`detection/opencv_service.py`
- `apps/mac/mac-app/Sources/PediatricsRAGMacApp/*.swift` 中的核心状态与接口文件

## 11. 最终约定

本仓库后续默认采用以下约定：

- 注释内容统一使用中文
- Python 文件级说明统一使用模块 docstring
- Python 公共接口优先使用 Google 风格 docstring
- Swift 公共类型和重要方法统一使用 `///`
- 文件头讲职责与边界，参数说明写在函数或方法注释中

如果后续发现某个子模块有更适合自己的细化规则，可以在子模块 README 中补充，但不得与本规范的核心原则冲突。
